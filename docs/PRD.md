# 面试通（mianshitong）PRD（MVP v0）

定位：你的专属 AI Agent 面试官。专注编程领域，尤其前端开发；提供简历优化、模拟面试、面试题解答等全方位面试辅导服务。

## 1. 背景与目标

### 背景

面向前端求职者的模拟面试，常见痛点是：题目零散、缺少追问、缺少“像面试官一样”的反馈（维度化评价、纠错与更优解）、复盘成本高。

### 产品目标（MVP）

- 让用户在 10-30 分钟内完成一场“前端岗位”模拟面试（问答 + 追问）。
- 面试结束生成结构化报告：维度评分、证据、改进建议、补齐知识点、下一步练习清单。
- 为后续上线到国内服务器、接入 DeepSeek 模型打好“可插拔模型 Provider”的工程基础。

### 非目标（MVP 不做）

- 简历优化全流程（先预留模块与数据结构，后续做）
- 面试题解答的完整知识库/检索增强（先做少量题库与结构化答案要点）
- 语音面试/实时语音转文字
- 完整账号体系/付费系统
- 复杂题库后台、多人协作
- 在线代码判题（可作为后续迭代）

## 2. 用户画像与场景

### 用户画像

- 目标：前端求职者（实习/初/中/高），希望短期提升面试表现。
- 典型行为：刷题 -> 模拟面试 -> 复盘 -> 针对性补齐短板。

### 核心场景

1. 用户选择岗位与方向，开始面试。
2. AI 面试官按节奏提问（可追问、可切题、可控难度）。
3. 用户答题后即时给反馈（可配置：每题后反馈/面试结束后统一反馈）。
4. 面试结束生成报告，并给出下一步学习与练习建议。

## 3. MVP 范围（功能需求）

### 3.0 产品模块（长期规划）

- 模拟面试（Interview Agent）：对话式面试 + 追问 + 报告（MVP 优先）
- 简历优化（Resume Agent）：针对岗位 JD 给出结构化修改建议、亮点提炼、项目描述改写（后续）
- 面试题解答（Q&A Agent）：针对单题给出分层讲解、常见误区、延伸追问与练习建议（后续）

### 3.1 面试配置

- 输入项：
  - 岗位方向：前端通用 / React / Vue / 工程化 / 性能 / 网络 / 安全 / Node（可选）
  - 级别：初级 / 中级 / 高级
  - 时长：10 / 20 / 30 分钟（或题数：5/8/10）
  - 反馈模式：每题反馈 / 结束后反馈
- 约束：
  - 需要保证“覆盖面”，避免全程只问单一主题。

### 3.2 对话式面试（核心）

- 角色设定：严格但友好、追问导向、关注工程实践与表达清晰度。
- 行为：
  - 首问：按配置挑选题目并开场
  - 追问：依据用户回答的缺口/亮点进行追问（1-2 轮）
  - 切题：如果超时或偏题，收束并进入下一题
  - 终止：到达时长/题数后结束，进入总结

### 3.3 评分与反馈

- 维度化评分（详见 `docs/Rubric.md`）：
  - 正确性、深度、结构化表达、工程化/实践、边界与权衡
- 反馈产物：
  - 优点（可复用表述）
  - 问题与风险点（带证据：引用用户回答要点）
  - 更优答案要点（分层：基础/进阶）
  - 建议追问与延展知识点

### 3.4 面试报告

- 结构：
  - 总体评分 + 各维度雷达（前端可视化后置）
  - 按题目汇总：题目、考点、你的回答摘要、评分、改进建议
  - 知识点清单：薄弱点（优先级）+ 推荐练习

### 3.5 题库（MVP）

- 先采用静态 JSON 题库（文件形式），每题包含：
  - id、标题、题干、方向标签、难度、考点列表、参考答案要点、常见误区、可追问列表
- 题库策略：
  - 以“可追问、可区分度”为优先；每题应能自然追问 1-2 次。

## 4. 体验与交互（信息架构）

### 页面（建议）

- `/`：配置页（方向/级别/时长/反馈模式）
- `/interview`：对话页（时间/进度、当前题目、输入框）
- `/report`：报告页（总结与建议）

### 关键交互

- 中途退出：提示保存本次会话（MVP 可仅本地保存）
- 反馈模式切换：每题反馈会影响对话节奏（需要在 prompt 中明确）

## 5. 结构化输出与接口契约（MVP 先定义）

### 5.1 “下一步提问”结构（模型输出建议）

模型每轮输出**结构化 JSON**，包含：

- `kind`: `"question" | "follow_up" | "evaluation" | "end"`
- `question`: 本轮要问的问题（自然语言）
- `intent`: 本题/追问的考察点（列表）
- `difficulty`: `"junior" | "mid" | "senior"`
- `time_budget_sec`: 建议用户回答时长
- `next_action`: `"wait_user" | "ask_follow_up" | "next_topic" | "end_interview"`

### 5.2 “评分反馈”结构（模型输出建议）

- `scores`: 各维度 1-5
- `evidence`: 从用户回答抽取的要点（不要长引用）
- `strengths` / `issues` / `better_answer_outline` / `learning_todos`

> 具体字段与评分口径见 `docs/Rubric.md` 与 `docs/PromptSpec.md`。

## 6. 技术与部署约束（面向上线）

### 6.0 技术栈（你已选定）

- Web/服务端：Next.js（TypeScript）
- 数据层：Prisma + PostgreSQL
- 部署：Docker（国内服务器）

### 6.1 模型接入（DeepSeek）

- 采用 Provider 抽象，避免代码与某一家模型 API 强绑定。
- 兼容策略（待确认）：
  1. 若 DeepSeek 提供 OpenAI-compatible 接口：按兼容模式实现
  2. 若为自定义接口：实现 `DeepSeekProvider` 适配器
- 需要你后续提供的信息（上线前必须确认）：
  - API Base URL、鉴权方式、模型名、限流/计费策略、是否支持 JSON Schema / 工具调用

### 6.2 国内服务器上线

- API Key 必须只保存在服务端环境变量，不下发前端
- 基础防护（MVP 可简化，但建议预留）：
  - 简单限流（按 IP/会话）
  - 日志脱敏（不记录用户隐私字段）
